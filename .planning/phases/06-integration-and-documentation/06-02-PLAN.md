- App Group: group.com.claudewidget
- CredentialStore: Keychain with kSecAttrAccessibleAfterFirstUnlock
- Data sharing: UserDefaults(suiteName: "group.com.claudewidget")
- Background: BGAppRefreshTask + WidgetKit TimelineProvider
- Widget: WidgetKit (StaticConfiguration, systemSmall + systemMedium)
- Min deployment: iOS 17.0
- Keychain accessGroup: nil by default, needs TEAM_PREFIX.com.claudewidget

Auth expiry flow (post Plan 01 fix):
- Android: Worker fetch 401 -> CredentialStore.clear() -> updateAll() -> widget shows "Sign in"
- iOS: Timeline fetch 401 -> CredentialStore.clear() -> re-check creds -> entry has hasCredentials=false -> widget shows "Sign in"

**1. Overview** (2-3 sentences)
- What the app does: Shows Claude.ai usage limits as a home screen widget
- Tech: Kotlin, Glance widgets, WorkManager background refresh

**2. Prerequisites**
- Android Studio (Ladybug or newer recommended)
- Android SDK with min API 26 (Android 8.0), target API 35
- JDK 17
- A Claude.ai account with an active session

**3. Building the App**
- Clone and open `android/` in Android Studio
- Sync Gradle (AGP 9.1.0, Gradle 9.3.1)
- Note: Do NOT apply kotlin-android plugin -- AGP 9.1.0 has built-in Kotlin support
- Build: `./gradlew assembleDebug`
- Install: `./gradlew installDebug` or use Android Studio Run

**4. First-Time Setup**
- Open the app after installation
- The app opens the Claude.ai login WebView
- Sign in with your Claude.ai credentials
- The app automatically captures your session cookie and organization ID
- After successful login, the app shows a confirmation screen
- The background worker starts immediately to fetch usage data

**5. Adding the Widget to Your Home Screen**
- Long-press on an empty area of the home screen
- Tap "Widgets" (or drag from the widget drawer)
- Find "Claude Usage" in the list
- Drag the widget to your desired position
- The widget supports two sizes: Small (shows usage bars) and Medium (adds updated time and refresh button)
- The widget auto-refreshes every 15 minutes via WorkManager

**6. Widget States**
- **Normal**: Shows 5-hour and 7-day usage with color-coded progress bars (green < 70%, orange 70-90%, red >= 90%)
- **Loading**: Shows "Loading..." while fetching data for the first time
- **Stale**: Shows "(stale)" in the title when cached data is older than 2 hours
- **Not logged in**: Shows "Sign in to Claude app" when no credentials are stored

**7. Session Expiry and Re-Login**
- Claude.ai sessions expire periodically
- When the background worker detects a 401/403 response, it automatically:
  1. Clears stored credentials
  2. Posts a "Claude session expired" notification
  3. Updates the widget to show "Sign in to Claude app"
- To re-login: tap the widget or the notification to open the app, which will present the login WebView again
- After re-login, the widget resumes normal operation automatically

**8. Troubleshooting**
- Widget shows "Loading..." indefinitely: Check if you're logged in by opening the app
- Widget not updating: Verify WorkManager is running (`adb shell dumpsys jobscheduler | grep claudewidget`)
- "Sign in" after fresh install: Normal -- open the app first to complete login

**9. Architecture Notes** (brief)
- Credentials: EncryptedSharedPreferences (AES256_GCM)
- Cached data: DataStore Preferences (shared between app and widget via same process)
- Background: WorkManager 15-min periodic with retry on failure
- Widget: Glance AppWidget with Responsive SizeMode (small + medium)

Create `ios/SETUP.md` with the following sections. This is a higher-level guide than README-XCODE-SETUP.md and references it for detailed Xcode steps.

**1. Overview** (2-3 sentences)
- What the app does: Shows Claude.ai usage limits as a home screen widget
- Tech: Swift, WidgetKit, BGAppRefreshTask background refresh

**2. Prerequisites**
- macOS with Xcode 15+ installed
- iOS 17.0+ device or simulator
- Apple Developer account (for App Groups on physical devices)
- A Claude.ai account with an active session

**3. Xcode Project Setup**
- The source files are in `ios/` but the Xcode project must be created manually (`.xcodeproj` is not committed)
- Follow `README-XCODE-SETUP.md` for step-by-step Xcode project creation
- Summary of what README-XCODE-SETUP.md covers:
  1. Create the Xcode project (ClaudeUsage, com.claudewidget, iOS 17.0)
  2. Add Widget Extension target (ClaudeUsageWidget, com.claudewidget.widget)
  3. Replace Xcode-generated files with the repo source files
  4. Configure App Groups on both targets

**4. App Groups Configuration** (critical for widget data sharing)
- App Group ID: `group.com.claudewidget`
- Must be enabled on BOTH targets: ClaudeUsage (main app) and ClaudeUsageWidget (extension)
- Steps: Target > Signing & Capabilities > + Capability > App Groups > add `group.com.claudewidget`
- On physical devices: also configure in Apple Developer Portal (Identifiers section)
- The widget reads cached usage data from `UserDefaults(suiteName: "group.com.claudewidget")`
- If App Groups is misconfigured, the widget will always show "Loading..." because it can't read shared data

**5. Keychain Sharing** (for credential access from widget extension)
- CredentialStore uses Keychain with `kSecAttrAccessibleAfterFirstUnlock`
- The `accessGroup` in `CredentialStore.swift` is `nil` by default
- For the widget extension to read credentials, set `accessGroup` to `"$(AppIdentifierPrefix)com.claudewidget"` (Xcode resolves the team prefix at build time)
- Enable Keychain Sharing capability on both targets with the same keychain group
- Without this, the widget will always show "Sign in to Claude app" because it can't read credentials

**6. Info.plist Configuration**
- Add `com.claudewidget.refreshUsage` to `BGTaskSchedulerPermittedIdentifiers` array in the main app's Info.plist
- Without this, background refresh will silently fail (no error, just no refreshes)

**7. Building and Running**
- Select the ClaudeUsage scheme
- Build and run on simulator or device
- To test the widget: also run the ClaudeUsageWidget scheme at least once
- Command line: `xcodebuild -scheme ClaudeUsage -destination 'generic/platform=iOS Simulator' build`

**8. First-Time Setup**
- Open the app after installation
- The app presents the Claude.ai login WebView
- Sign in with your Claude.ai credentials
- The app uses JavaScript injection to capture your organization ID from API requests
- Session cookies are extracted from WKWebView's cookie store
- After login, credentials are stored in Keychain and the first data fetch begins

**9. Adding the Widget to Your Home Screen**
- Long-press on the home screen
- Tap the "+" button in the top-left corner
- Search for "Claude Usage" or scroll to find it
- Choose Small or Medium size:
  - Small: Shows usage bars and reset time
  - Medium: Adds last-updated time and refresh button
- Tap "Add Widget" and position it

**10. Widget States**
- **Normal**: 5-hour and 7-day usage with color-coded progress bars (green < 70%, orange 70-90%, red >= 90%)
- **Loading**: Shows "Loading..." while fetching data for the first time
- **Stale**: Shows "(stale)" in the title when cached data is older than 2 hours
- **Not logged in**: Shows "Sign in to Claude app" when no credentials are stored

**11. Session Expiry and Re-Login**
- Claude.ai sessions expire periodically
- When the timeline provider detects a 401/403 response:
  1. Credentials are cleared from Keychain
  2. A local notification is posted ("Claude session expired")
  3. The widget immediately transitions to "Sign in to Claude app"
- To re-login: tap the widget to open the app, sign in again
- The widget will resume normal operation on the next timeline refresh after re-login

**12. Background Refresh**
- Uses two mechanisms:
  1. WidgetKit TimelineProvider: Fetches data during timeline reloads (every 15 min)
  2. BGAppRefreshTask: Schedules background fetches when the app is not in the foreground
- The BGAppRefreshTask chain re-schedules itself before completing (continuous refresh loop)
- iOS may defer background tasks based on system conditions (battery, network, usage patterns)

**13. Troubleshooting**
- Widget shows "Sign in" but you are logged in: Check Keychain Sharing is enabled on both targets with matching group
- Widget shows "Loading..." indefinitely: Verify App Groups is configured on both targets with `group.com.claudewidget`
- Background refresh not working: Confirm `BGTaskSchedulerPermittedIdentifiers` contains `com.claudewidget.refreshUsage` in Info.plist
- Widget not appearing in widget gallery: Ensure the widget extension target builds successfully

**14. Architecture Notes** (brief)
- Credentials: Keychain (kSecAttrAccessibleAfterFirstUnlock, shared via access group)
- Cached data: UserDefaults via App Groups (shared between app and extension)
- Background: BGAppRefreshTask chain + WidgetKit TimelineProvider
- Widget: WidgetKit StaticConfiguration (systemSmall + systemMedium)
